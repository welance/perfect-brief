# Contributing a rule

Anyone can. A good rule PR contains:

1. **A rule file** in `rules/` matching `rule.schema.json`:
   - `criteria` — the human-language pass condition the model applies.
   - `pass_examples` / `fail_examples` — calibration anchors *and* regression cases.
   - `weight` + `severity` — how much it pulls, and its cap. Code-side; the model
     never sees these.
   - `compliment` + `suggestion` — what the submitter reads on pass / on miss.
   - `references` — verifiable sources, **authored and checked by you in the PR**
     (not generated by a model). Tier them: `standard` > `practice` >
     `encyclopedic`. Pin the exact version/clause. Encyclopedic (e.g. Wikipedia)
     may not be the sole anchor of a normative claim.

2. **Fixtures** in `fixtures/` if the rule changes scoring: at least one brief
   that exercises it, with an expected band. CI must stay green.

3. **A rationale** in the `rationale` field: why this predicts a better brief or
   a smoother build. "We should weight X higher" is a one-line `weight` change —
   say why.

Run `python evaluate.py` before opening the PR. Red corpus = the PR isn't ready.

## Principle: substance over polish

In the AI era, fluent prose is free — a brief can be smooth and empty. Anchor
every criterion to a verifiable signal (a testable condition, a number, a named
source) so the score rewards substance, not style.
