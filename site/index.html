what it does

It reads a brief and returns two separate things. A score — a weighted average over fourteen public rules (is there a real problem statement? named users? a measurable outcome? a definition of done?) with a plain band label. And a gate — four hard requirements that must hold before a brief may publish at all: a clear title, a stated problem, a budget above the engagement floor, and full anonymisation.

Keeping them separate is the point. A brilliant brief that leaks the client's name shouldn't go on a blind noticeboard — and no average should be able to paper over that.

how it works

There is one seam, and everything depends on it. The ruleset is a folder of versioned YAML files — anyone can read them, anyone can propose a change. A language model reads one rule at a time and returns a verdict: pass, partial, or fail, with a quote from the brief as evidence. That is all it does. Plain, deterministic code — which the model never sees — owns every weight, the gate, and the decision.

The judge runs at temperature zero, so the same brief against the same ruleset version gives the same score, reproducibly. A corpus of labelled example briefs runs in CI: a rule change that moves the numbers has to move those fixtures too, in the open, in the same commit.

curl -X POST https://briefs.welance.com/v1/score \
  -H 'content-type: application/json' \
  -d '{"brief":"…","judge":"llm"}'

{ "score": 92.0, "band": "Directory-ready", "decision": "blocked",
  "gate": { "passed": false, "missing": ["anonymised"] },
  "verdicts": [ … per-rule status + verbatim evidence … ],
  "ruleset_version": "1.0.0+83107baec655" }

The ruleset_version pins exactly which bar judged a brief — an audit trail you can re-run. The console works offline too, on a keyword mock of the judge, so you can try the mechanics without an API key.

why we built it, honestly

We are Welance, a small collective of independent senior teams, working together since 2012. We're building welance/Directory — a curated, blind noticeboard where cross-functional teams pick up product briefs on the strength of the brief, not the brand behind it. For that to work, briefs need a bar: readable, consistent, and blind-safe.

There's an obvious conflict in one company operating both the noticeboard and the bar. Our answer is to make the bar open. The rules are public files with public weights; the €10k budget floor is a stated policy, not a hidden filter; the test corpus that guards changes is public; and the whole thing is MIT-licensed, so if you disagree with our bar you can fork it and run your own. We think that's the only honest way to run one.

This tool is the distilled version of what fifteen years of handoffs taught us: the products that go well aren't the ones with the most polished decks — they're the ones where a small, senior, cross-functional team got a brief honest enough to start from. We can't open-source the teams. We can open-source the bar.

Together, this service and the Directory are the digitalised version of what Welance has always done in person — matching honest briefs with senior teams — rebuilt with the approaches we now believe in: open rules instead of opinions, reproducible scores instead of gut feel. Two convictions drive it. Writing a good technical brief for a software product should be easy — the bar should help you clear it, not just measure you. And no one should have to commit to the first team they meet: a brief worth building deserves to meet several, so the fit — technical and cultural — is chosen, not defaulted to.
